current context:
- containr (formerly znskr) is a paas managing docker containers, pingora proxy routing, and acme certs.
- metadata uses sqlite via sqlx (wal mode, relational schema); sled is supported as legacy.
- dashboard is a solidjs spa (pnpm/node) with a dark theme; cli is `containrctl`.
- managed services include postgres (with pitr/pgdog), valkey, mariadb, qdrant, and rabbitmq.
- recent refactors centralized service lifecycle, separated the cli, added persistent mounts, and verified routing with live docker e2e tests.

important decisions:
- service model is authoritative: image-only apps, cron jobs, and background workers are first-class runtime types.
- no auto-migration for legacy sqlite json blobs; database runs in relational mode.
- single-container deployments fall back to replicated `web` service if scaled > 1.
- acme cert renewal runs entirely in-binary with pingora reloading certs from disk without restarts.
- deployment logs run strictly inside bounded requests, decoupled from live streaming.
- keep sqlite as the default metadata backend, though the public database trait abstracts access.

architectural notes:
- internal network isolates database/service traffic, per-app network limits application scopes.
- pingora handles both api and web frontend routing seamlessly, forwarding the base dashboard hostname directly to the api.
- rustfs buckets use server-managed credentials; metadata tracks endpoints (management, internal, public).
- frontend dependencies rigidly use repo-pinned node/pnpm.

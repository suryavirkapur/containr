0.1.14-alpha - 2026-03-09
- add a canonical `/api/services` surface that lists app services, managed
  databases, and managed queues through one backend inventory model with
  shared status, network, grouping, and lifecycle metadata
- add unified service lifecycle handlers plus `containrctl services` commands
  for list/get/logs/start/stop/restart/delete, including app-service runtime
  control without forcing a full project redeploy
- add `crates/containr-runtime/src/app_service_manager.rs` so single deployed
  app services can be started, stopped, restarted, and tailed through the same
  docker runtime logic outside the deployment worker
- add `scripts/e2e-services.sh`, a checked-in docker-backed backend harness
  that validates deploy, wildcard routing, unified service inventory, app and
  managed-service lifecycle, public ports, grouped valkey reachability, and
  teardown cleanup end to end
- change the default `containrctl` instance url to `http://127.0.0.1:2077`
  and update readme/install/configuration/quickstart docs to match the current
  service-centric cli and api defaults
- make the service group the docker network boundary for backend runtime:
  grouped managed services now join the project network, ungrouped managed
  services get their own private `containr-svc-*` network, and group deletion
  now tears down grouped managed services before removing the project network
- promote qdrant and mariadb to first-class managed database types in the api
  and cli, and expose normalized `service_type`, `group_id`, `network_name`,
  `public_ip`, and public connection metadata in managed service responses
- add embedded sqlite migration files plus an in-binary migration runner so
  schema creation no longer lives in a single inline sql batch
- add `containrctl` support for `groups` as a projects alias, managed-service
  group filters, database public exposure, postgres pitr toggling, postgres
  pgdog toggling, and queue public exposure
- add `containrctl databases base-backup`, `restore-point`, and `recover`
  commands so postgres pitr flows can be exercised without dropping to raw
  http calls
- extend `scripts/e2e-services.sh` with live postgres and pgdog coverage for
  direct access, proxy access, base backups, restore points, recovery, public
  postgres ports, and the invariant that each postgres service keeps exactly
  one pgdog frontend across restarts
- fix postgres pitr recovery after live testing by normalizing base-backup
  permissions so the host-side recovery flow can read container-created backup
  directories instead of failing with `permission denied`
- fix postgres restore-point recovery by forcing a wal switch and copying the
  live `pg_wal` segment files into the mounted local archive before stop and
  restore, and normalize the copied wal permissions so recovery can actually
  read those files instead of failing with `permission denied`
- refactor `crates/containr-runtime/src/database_manager.rs` by moving the
  postgres pitr backup, restore, wal sync, and recovery helpers into
  `crates/containr-runtime/src/database_manager/pitr.rs` so runtime container
  lifecycle code is separated from the postgres recovery flow
- refactor `crates/containr/src/client.rs` so repeated resource list/get/logs/
  start/stop/restart/delete path handling for services, databases, and queues
  now flows through shared helper functions instead of duplicating the same
  string-building logic in each command matcher
- raise the default `containrctl` request timeout to 180 seconds so postgres
  recovery and other long-running control-plane actions do not fail client-side
  while the server is still doing valid work
- change the default api port from `3000` to `2077`, expose dashboard/base
  domain details through settings, and update the example config, install
  script, install docs, and readme for the new defaults and wildcard routing
- fix qdrant provisioning by removing the broken image health probe that left
  healthy qdrant containers stuck in failed create flows
- live-test the new backend model against real docker: same-group valkey
  access succeeded with `PONG`, cross-group access failed by dns isolation,
  qdrant public http access worked, mariadb public tcp access worked, postgres
  pgdog access worked, and wildcard service subdomains routed through the proxy
- note one remaining rough edge: `containrctl databases list --group-id ...`
  returns `404` after the referenced group has been deleted because the filter
  validates that the target group still exists
- add image-only project deployments via `deployment_source = none`, plus
  rust cron-job scheduling for `cron_job` services without source checkout
- narrow the first-class managed service surface to postgres, redis, and
  rabbitmq in the api and cli
- expand `containrctl` with toml-based project apply/deploy/rollback/delete
  flows plus managed database and queue lifecycle commands
- fix rollback for multi-service image deployments by accepting per-service
  image artifacts instead of requiring a single top-level deployment image id
- mark superseded deployments as `stopped` after successful rollouts so
  deployment history matches the actual docker state
- enforce redis passwords in valkey containers with an explicit
  `valkey-server --requirepass ...` startup command
- make managed database logs return bounded output instead of hanging forever
  by accidentally following the docker log stream
- update database roundtrip tests to assert the current service-first routing
  model instead of deprecated project-level domain behavior
- make sqlite the default metadata backend in generated config, examples, and docs while keeping sled available as an opt-in backend
- add a dedicated vps install script that installs docker from the official apt repository, downloads the latest containr release, writes production config, and installs the systemd service
- add a `gh` release script that builds release assets, packages `containr` and `containrctl`, and publishes a prerelease from the current version
- update the readme and deployment docs to use the current `suryavirkapur/containr` repository, release assets, and client cli workflow
- move the dashboard homepage to the unified `/api/services` inventory so the
  landing page now shows app services, managed databases, and managed queues
  together instead of only project-scoped app services
- add homepage quick-create tiles for web services, private services,
  background workers, cron jobs, postgres, valkey, mariadb, qdrant, and
  rabbitmq so service creation is visible from `/` instead of buried in menus
- let `web/src/pages/NewApp.tsx` honor `?service_type=` presets, including
  cron jobs, so dashboard quick-create links open the app-service form with the
  requested runtime shape already selected
- let `web/src/pages/Databases.tsx` and `web/src/pages/Queues.tsx` honor
  `?create=1&type=...` so the homepage can deep-link directly into the correct
  managed-service create flow
- fix the database create form to submit `redis` for valkey instead of the
  unsupported `valkey` value, and remove the stale unsupported `nats` option
  from the queue create ui
- fix the existing `web/src/pages/AppDetail.tsx` jsx tree mismatch so the
  solid/vite frontend now builds successfully again
- regroup the dashboard inventory by service group so `/` now renders services
  inside their project/group boundary instead of one flat cross-instance list,
  with standalone services isolated into their own sections
- add frontend restart controls directly to the grouped service inventory on
  the dashboard so operators can restart any listed service without opening a
  detail page first
- replace the old dashboard quick-create deep links to database/queue modals
  with one unified full-page create flow at `/projects/new` for app services,
  managed databases, and managed queues
- extend `web/src/pages/NewApp.tsx` into a multi-mode service creator that can
  create app services, managed databases, and rabbitmq queues, and lets
  managed services attach to an existing group or stay standalone
- change `web/src/pages/Databases.tsx` and `web/src/pages/Queues.tsx` so they
  no longer own modal create flows; they now link creation into the shared
  `/projects/new` interface and expose restart actions from the list pages

0.1.0-alpha - 2026-01-15
------------------------

initial alpha release

core features:
- docker container deployment from github repositories
- automatic containerfile/dockerfile detection and building
- pingora reverse proxy with http/https routing
- acme/let's encrypt automatic ssl certificates
- sled embedded database for persistent state

protocol support:
- websocket passthrough for real-time applications
- grpc (http/2) passthrough support
- server-sent events (sse) streaming support

multi-container apps:
- container services with replica scaling
- service dependencies (depends_on)
- health checks with configurable intervals
- restart policies (never, always, onfailure)
- memory and cpu limits per service

environment and secrets:
- environment variables per app
- secret masking (values hidden in api responses)
- encrypted storage for sensitive values

managed services:
- managed databases (postgres, mysql, redis)
- managed queues (redis, rabbitmq)
- storage buckets

dashboard:
- solidjs dashboard with dark theme
- real-time container logs via websocket
- deployment history and status
- certificate management ui

authentication:
- jwt-based authentication
- github oauth integration
- user registration and login

api:
- restful api for all operations
- openapi documentation at /api/docs
- webhook support for github push events

documentation:
- docs/quickstart.txt: 5-minute quickstart guide
- docs/architecture.txt: system architecture overview
- docs/api-reference.txt: complete api documentation
- docs/configuration.txt: detailed config options
- docs/deployment-guide.txt: production deployment guide
- docs/troubleshooting.txt: common issues and solutions

known limitations:
- linux x86_64 only
- requires root for ports 80/443 (or use custom ports)
- acme staging mode recommended for testing

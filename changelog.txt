0.1.10-alpha - 2026-03-08
- start managed database containers during database creation and persist failed provisioning attempts as failed database resources
- redesign the database detail page into overview/connect/access/recovery/backups/container tabs with a denser service summary and quick-connect panel
- mask database passwords and connection strings by default until explicitly shown, while keeping dedicated copy actions for each secret and endpoint
- navigate to the new database detail page after a successful create and refresh the databases list when provisioning fails
- add a bundled svg favicon for the dashboard frontend
- sync the frontend pnpm lockfile with the checked-in package dependencies so release builds succeed with frozen lockfiles
- add `mise`-driven `deploy-prod` and `release-prod` tasks that build from a
  clean worktree commit, upload the release binary, restart prod, and verify
  the dashboard endpoints
- bootstrap the oldest existing user to admin on startup if no admin exists,
  make the first new account admin, and restrict global settings plus system
  stats to admin users
- scope local database backups under per-database directories keyed by database
  id instead of a shared name-prefixed folder, so same-name databases cannot
  collide
- start managed queues during queue creation and persist failed queue startup
  attempts as failed resources instead of leaving pending metadata behind
- mark interrupted in-progress deployments as failed during startup recovery so
  restarts do not leave stale pending/building/starting rows behind
- hide the dashboard system monitor for non-admin users once host stats require
  admin access
- enable the workspace `uuid` v5 feature in the committed manifest so the clean
  release build matches the checked-in code and `mise run release-prod` does
  not depend on unstaged local manifest changes

0.1.9-alpha - 2026-03-07
- fix the app detail container monitor for multi-replica apps by refreshing the container list while deployments are live and labeling replicas distinctly in the api as `service #n`
- remove the legacy single-container deployment branch by normalizing apps into a default `web` service, routing legacy running deployments through the new service model until redeploy, and rejecting empty explicit service lists
- add explicit per-service public http selection for multi-service apps, persist it in sqlite, expose it in the api/ui, and fall back to `web` or the first service so older apps still route correctly
- make the dashboard's single-app instance count actually work by converting legacy single-container apps into a replicated `web` service when more than one instance is requested
- add service-level persistent app mounts with sqlite persistence, docker bind mounts, mount validation, and ui/api support for multi-service apps
- add per-service private registry credentials with encrypted persistence, authenticated image pulls during deployment, and ui/api support for image-based services
- add browser terminal access for running containers with a docker exec websocket bridge and an xterm-based frontend tab in the container monitor
- add per-service command, entrypoint, and working-directory overrides with sqlite persistence, deployment-time docker wiring, api validation, and advanced ui inputs for multi-service apps
- add explicit per-service additional container ports with sqlite persistence, docker exposed-port wiring, api validation, and ui support
- add service-mount backup and restore flows with tar archive endpoints, archive path validation, and app detail controls that work even when a service container is stopped
- rename the project from znskr to containr across crate names, binary/config names, example files, docs, ui copy, and internal docker naming
- split rustfs storage configuration into management, internal docker, and optional public proxy endpoints with bucket connection details exposed through the api and ui
- allow `containr.toml` deployments to override encryption through `CONTAINR_ENCRYPTION_KEY` while still accepting the legacy `ZNSKR_ENCRYPTION_KEY` during rollout
- route the base dashboard hostname through the api upstream for both ui and `/api` traffic so the renamed proxy deployment serves the dashboard correctly
- connect app containers to the shared `containr-internal` docker network before start so short-lived or restarting processes do not fail deployment during post-start network attachment
- replace bun-based frontend tooling with mise-managed node 24.14.0 and pnpm 10.30.3 across the repo build scripts and docs
- remove the local git push management and smart-http git endpoints from the api, proxy, ui, docs, and generated client schema
- add a database backend trait with config-selectable sled and sqlite implementations behind the shared database api
- enable sqlite metadata storage in wal mode with schema-managed tables and ordered deployment log persistence
- redesign the sqlite backend into typed relational tables for apps, env vars, domains, services, deployments, managed services, and github app installations instead of json blob rows
- hydrate nested app, service deployment, and github app installation relations from sqlite while keeping the public database api unchanged
- reject legacy sqlite `key/value(json)` tables at startup instead of auto-migrating them into the relational schema
- switch runtime database startup and examples to read the backend selection from the toml config
- fix github app and github oauth token decryption in deployment source resolution so stored secrets use the shared derived-key helper instead of raw config bytes
- preload saved deployment logs before opening the live build stream, and resume websocket log streaming from a supplied offset
- stop creating orphan pending deployments when source resolution or deployment queueing fails, and surface deploy trigger errors in the app detail ui
- fix github webhook deployments so failed source resolution or failed queue sends also clean up the pending deployment row
- stop swallowing github app config database read failures during deployment source resolution, so broken metadata reads fail fast instead of falling back silently
- paginate github app installation and repository listings so the repo picker and deploy token lookup see all accessible repos
- add periodic in-binary acme renewal passes for the dashboard domain and all managed custom domains
- persist stored certificates back to disk on startup and refresh pingora's tls cache when cert files change
- record certificate expiry from the issued leaf certificate instead of assuming a fixed 90 day lifetime
- fix managed service networking so apps, databases, and queues share a stable internal docker network
- add docker network aliases for app services and normalize managed service internal hostnames
- remove fixed host port binding for managed queues to avoid port collisions across instances
- pull missing managed database and queue images automatically before first container start
- add configurable public exposure for managed queues and optional custom external ports for both queues and databases
- let database and queue exposure settings persist without forcing stopped services to start
- add optional rustfs-backed database backup uploads that verify the uploaded object before reporting success
- switch storage buckets to server-managed rustfs credentials instead of misleading per-bucket client credentials
- add postgres-only point in time recovery for managed databases with wal archiving, local base backups, restore points, and recovery to a target time or restore point
- add optional pgdog frontends for managed postgres databases with api and ui controls for internal and public exposure
- persist managed database pitr/proxy metadata in sqlite and expose the new fields through the generated openapi client
- unify deployment job source resolution across github app, github oauth fallback, manual deploys, webhooks, and git push
- fix local-path deployment sources so validation and git-backed local repos clone without shallow-fetch errors
- fix manual deployments for filesystem-backed repositories so they resolve as local sources instead of remote git
- wait for managed queue and database health checks before reporting them as running
- make git post-receive hooks fail when deployment triggering fails instead of silently succeeding
- update dashboard copy from github-only wording to generic repository/source wording
- expose queue passwords in queue api responses and sync checked-in openapi artifacts
- add rustfs access/secret config fields, refresh the storage ui for server-managed buckets, and expose the new backup/exposure options in the ui

0.1.0-alpha - 2026-01-15
------------------------

initial alpha release

core features:
- docker container deployment from github repositories
- automatic containerfile/dockerfile detection and building
- pingora reverse proxy with http/https routing
- acme/let's encrypt automatic ssl certificates
- sled embedded database for persistent state

protocol support:
- websocket passthrough for real-time applications
- grpc (http/2) passthrough support
- server-sent events (sse) streaming support

multi-container apps:
- container services with replica scaling
- service dependencies (depends_on)
- health checks with configurable intervals
- restart policies (never, always, onfailure)
- memory and cpu limits per service

environment and secrets:
- environment variables per app
- secret masking (values hidden in api responses)
- encrypted storage for sensitive values

managed services:
- managed databases (postgres, mysql, redis)
- managed queues (redis, rabbitmq)
- storage buckets

dashboard:
- solidjs dashboard with dark theme
- real-time container logs via websocket
- deployment history and status
- certificate management ui

authentication:
- jwt-based authentication
- github oauth integration
- user registration and login

api:
- restful api for all operations
- openapi documentation at /api/docs
- webhook support for github push events

documentation:
- docs/quickstart.txt: 5-minute quickstart guide
- docs/architecture.txt: system architecture overview
- docs/api-reference.txt: complete api documentation
- docs/configuration.txt: detailed config options
- docs/deployment-guide.txt: production deployment guide
- docs/troubleshooting.txt: common issues and solutions

known limitations:
- linux x86_64 only
- requires root for ports 80/443 (or use custom ports)
- acme staging mode recommended for testing
